# Human-in-the-Loop (HITL) для Foundry Pipeline

**Статус**: Реалізовано  
**Область**: Foundry orchestrator, TUI monitor, протокол агентів, Telegram Q&A бот

---

## Огляд

Система HITL дозволяє агентам пайплайну призупинятися та задавати питання, коли вони стикаються з неоднозначністю, замість того щоб завершуватися з помилкою або вгадувати. Пайплайн може опціонально продовжувати роботу з іншими агентами, поки очікує відповідей.

---

## Як це працює

### 1. Агент задає питання

Коли агент не може продовжити роботу без людського введення, він:

1. Записує питання до `tasks/<slug>--foundry/qa.json`
2. Оновлює свою секцію в `handoff.md` зі статусом `waiting_answer`
3. Завершується з кодом `75` (EX_TEMPFAIL)

Оркестратор виявляє код виходу 75 і переводить задачу в стан `waiting_answer`.

### 2. Людина відповідає

Три способи відповісти:

**TUI Monitor** (рекомендовано):
```bash
./agentic-development/foundry.sh
# Натисніть Enter на задачі, що очікує, щоб відкрити Q&A view
```

**CLI (неінтерактивний)**:
```bash
./agentic-development/foundry.sh answer <slug> --question q-001 --answer "Використати edge-auth"
```

**Telegram бот** (якщо налаштовано):
- Бот надсилає inline keyboard з варіантами відповідей
- Натисніть кнопку або введіть текстову відповідь

### 3. Відновлення пайплайну

Після відповіді на всі блокуючі питання:

```bash
./agentic-development/foundry.sh resume-qa <slug>
```

Або через TUI: натисніть `Ctrl+Enter` у Q&A view.

---

## CLI команди

| Команда | Опис |
|---------|------|
| `foundry.sh waiting` | Список всіх задач, що очікують відповідей |
| `foundry.sh answer <slug>` | Відкрити TUI Q&A view для задачі |
| `foundry.sh answer <slug> --question <id> --answer <text>` | Відповісти через CLI |
| `foundry.sh resume-qa <slug>` | Відновити пайплайн після відповіді |
| `foundry.sh status` | Показує кількість `waiting_answer` |
| `foundry.sh telegram-qa start` | Запустити Telegram Q&A бот |
| `foundry.sh telegram-qa stop` | Зупинити Telegram Q&A бот |

## Формат Q&A файлу (`qa.json`)

```json
{
  "version": 1,
  "questions": [
    {
      "id": "q-001",
      "agent": "u-architect",
      "timestamp": "2026-03-26T14:30:00Z",
      "priority": "blocking",
      "category": "clarification",
      "question": "Яку систему автентифікації слід змінити?",
      "context": "Знайдено в docker/traefik/dynamic.yml та brama-core/src/Security/",
      "options": [
        "edge-auth (Traefik middleware)",
        "internal JWT auth (brama-core)",
        "обидві"
      ],
      "answer": null,
      "answered_at": null,
      "answered_by": null
    }
  ]
}
```

### Пріоритет питань

| Пріоритет | Поведінка |
|-----------|-----------|
| `blocking` | Пайплайн призупиняється. Необхідно відповісти перед відновленням. |
| `non-blocking` | Пайплайн може продовжуватися. Агент буде перезапущений після отримання відповідей. |

### Категорії питань

- `clarification` — Неоднозначний опис задачі
- `preference` — Стилістичний або підхідний вибір
- `approval` — Потребує явного підтвердження людини
- `technical` — Технічне рішення зі значним впливом

## TUI Q&A View

Коли задача перебуває в стані `waiting_answer`, натискання `Enter` у списку задач відкриває редактор Q&A:

```
┌─ Q&A: task-slug ─────────────────────────────────────────────────┐
│                                                                   │
│  ┌─ Питання (u-architect) ──────────┐  ┌─ Відповідь ───────────┐ │
│  │  ► Q1 [blocking] *              │  │  Введіть відповідь...  │ │
│  │    Яку систему auth змінити?     │  │                        │ │
│  │    • edge-auth (Traefik)        │  │                        │ │
│  │    • internal JWT (brama-core)  │  │                        │ │
│  │    • обидві                     │  │                        │ │
│  │                                  │  │                        │ │
│  │    Q2 [non-blocking]            │  │                        │ │
│  │    Конвенція іменування REST?    │  │                        │ │
│  └──────────────────────────────────┘  └────────────────────────┘ │
│                                                                   │
│  * = без відповіді    ► = вибрано                                 │
│  ↑↓ Навігація  Tab перемикання панелі  1-9 швидкий вибір  Esc зберегти │
└───────────────────────────────────────────────────────────────────┘
```

**Клавіші:**

| Клавіша | Дія |
|---------|-----|
| `↑` / `↓` | Навігація між питаннями |
| `Tab` | Перемикання фокусу між списком питань та редактором відповіді |
| `1`-`9` | Швидкий вибір варіанту з пропозицій агента |
| `Esc` | Зберегти відповіді та повернутися до списку задач |
| `Ctrl+S` | Зберегти відповіді без виходу |

## Telegram Q&A Бот

Автономний Telegram бот забезпечує двосторонній Q&A без необхідності використання TUI.

### Налаштування

```bash
# 1. Встановити змінні середовища в .env.local
PIPELINE_TELEGRAM_BOT_TOKEN=<токен від @BotFather>
PIPELINE_TELEGRAM_CHAT_ID=<ID чату або групи>

# Опціонально: обмежити доступ до конкретних user ID
PIPELINE_TELEGRAM_ALLOWED_USERS=123456789,987654321

# 2. Встановити залежності (одноразово)
cd agentic-development/telegram-qa && npm install

# 3. Запустити бот
./agentic-development/foundry.sh telegram-qa start
```

### Процес роботи бота

1. Агент завершується з кодом 75 → оркестратор надсилає Telegram сповіщення
2. Бот надсилає питання з inline keyboard кнопками
3. Користувач натискає кнопку або вводить текстову відповідь
4. Бот записує відповідь до `qa.json`
5. Якщо всі блокуючі питання відповіді → бот викликає `foundry.sh resume-qa`

### Graceful Degradation

| Сценарій | Поведінка |
|----------|-----------|
| Токен Telegram не налаштовано | Сповіщення тихо пропускаються, тільки TUI Q&A |
| Бот не може запуститися (node відсутній) | Fallback на тільки сповіщення + TUI Q&A |
| Бот падає під час сесії | Відповіді, вже збережені в `qa.json`, зберігаються |

## Протокол агента

### Запис питань

Агенти записують до `qa.json` та завершуються з кодом 75:

```bash
# Приклад структури qa.json
{
  "version": 1,
  "questions": [{
    "id": "q-001",
    "agent": "u-coder",
    "timestamp": "2026-03-26T14:30:00Z",
    "priority": "blocking",
    "category": "clarification",
    "question": "В якій базі даних слід створити нову таблицю?",
    "context": "brama-core/src/Entity/ — дві бази даних: main та analytics",
    "options": ["основна база даних", "аналітична база даних"],
    "answer": null,
    "answered_at": null,
    "answered_by": null
  }]
}
```

### Коди виходу

| Код виходу | Значення |
|------------|----------|
| `0` | Успіх — агент завершився нормально |
| `75` | Очікування відповіді — агент має питання в `qa.json` |
| `124` | Таймаут |
| `1-74, 76-123` | Помилка |

### Відновлення

Коли агент відновлюється після відповідей, оркестратор додає до промпту:

```
You are RESUMING after human answered your questions.
Read qa.json for answers. Continue from where you stopped.
Your previous work is preserved in handoff.md and git history.
```

Агент ПОВИНЕН:
1. Прочитати `qa.json` — всі відповіді тепер заповнені
2. Продовжити роботу, враховуючи відповіді
3. НЕ перезадавати питання, на які вже є відповіді

## Машина станів

```
pending → in_progress → completed
                ↓              ↑
          waiting_answer ──────┘  (відновлення після відповідей)
```

### Поля `state.json` (під час очікування)

```json
{
  "status": "waiting_answer",
  "waiting_agent": "u-architect",
  "waiting_since": "2026-03-26T14:30:00Z",
  "questions_count": 2,
  "questions_answered": 0,
  "resume_from": "u-architect"
}
```

## Режим Continue-on-Wait

За замовчуванням пайплайн призупиняється, коли агент завершується з кодом 75. Щоб дозволити пайплайну продовжувати роботу з наступними агентами під час очікування:

```json
// pipeline-plan.json
{
  "continue_on_wait": true
}
```

Коли `continue_on_wait: true`:
- Пайплайн переходить до наступного агента
- Агент, що очікує, позначається для подальшого відновлення
- Кілька агентів можуть накопичувати питання одночасно

---

## Агент-до-агента вирішення

Перед ескалацією до людини, оркестратор намагається вирішити питання через `u-architect`:

1. Агент завершується з кодом 75
2. Оркестратор запускає `u-architect` в режимі Q&A-responder
3. Якщо `u-architect` може відповісти на всі блокуючі питання → пайплайн продовжується автоматично
4. Якщо питання залишаються → ескалація до людини

Відповіді агентів відрізняються від відповідей людини через `answer_source`:
- `"agent"` — відповів інший агент
- `"human"` — відповіла людина через TUI
- `"telegram"` — відповіла людина через Telegram
- `"cli"` — відповідь через `foundry.sh answer` CLI

---

## Зворотна сумісність

- Існуючі задачі без `qa.json` продовжують працювати нормально
- Агенти без протоколу HITL у своєму `.md` ніколи не завершаться з кодом 75
- Код виходу 75 раніше трактувався як загальна помилка — тепер має конкретне значення
- Старі версії monitor відображатимуть `waiting_answer` як невідомий стан (graceful degradation)
